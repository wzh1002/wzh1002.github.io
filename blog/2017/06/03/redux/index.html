Redux源码分析

发表于 2017-06-03 | 分类于 扯淡 | | 阅读次数
| 字数统计 6,825 | 阅读时长 32

  前段时间一直在学习React相关的技术栈，对Redux有了一定的了解。在这里，将会基于对Redux的理解以及官方文档介绍来先猜想Redux源码实现方式，再对源码进行对比分析来找出自己理解上的一些不足和遗漏，同时也学习下Redux的设计思路。

结构分析

  从GitHub上找到Redux官方托管的代码，进入到src目录下可以看到这么些文件。

  从目录上不难看出，Redux对外提供了5个方法(createStore、combineReducers、bindActionCreators、applyMiddleware、compost)。
  而则index.js负责从当前目录下将其他5个提供方法的js文件引入，然后统一由index.js对外导出。

createStore

个人猜想

  从官方API文档中可以得知，createStore(reducer, [preloadedState], enhancer)返回一个保存应用所有state的对象。

  • reducer (Function): 接收两个参数，分别是当前的 state 树和要处理的 action，返回新的 state 树。
  • [preloadedState] (any): 初始时的 state。在同构应用中，你可以决定是否把服务端传来的 state 水合（hydrate）后传给它，或者从之前保存的用户会话中恢复一个传给它。如果你使用 combineReducers 创建 reducer，它必须是一个普通对象，与传入的 keys 保持同样的结构。否则，你可以自由传入任何 reducer 可理解的内容。
  • enhancer (Function): Store enhancer 是一个组合 store creator 的高阶函数，返回一个新的强化过的 store creator。这与 middleware 相似，它也允许你通过复合函数改变 store 接口。

然后返回的对象中有这么几个方法

  • getState() 返回应用当前的state树
  • dispatch(action) 分发action，用来改变state树
  • subscribe(listener) 添加监听器，用来订阅state改变，返回一个unsubscribe函数
  • replaceReducer(nextReducer) 替换当前用来计算state的reducer(通过createStore传入的)

  从文档的描述中可以大致猜测出createStore中使用了闭包，并且内部有一个变量专门用来存储当前state。并且对外提供一个getState方法用来获取这个state。这也符合Redux的原则，state是只读的。
  然后再看dispatch，dispatch用来传递action给reducer，让reducer去改变应用树的状态。那么dispatch方法中必然有currentReducer(currentState, action)这样的调用。
  同时，createStore中实现了发布-订阅(不了解的可以自己搜下发布-订阅模式)。那么也必然有一个存放监听函数的数组，由于订阅的是当前state树的状态是否改变，而state树又只能由dispatch改变。显然，dispatch中还有对state改变事件的发布。
  接下来是replaceReducer，看上去直接用注入的nextReducer来替换以前的reducer即可。

源码及对比分析

  现在逐步分析源码，然后和原来猜想的实现进行对比。
  首先是原猜想中//do type check部分。和预料的一样，进行类型判断检查，由于preloadedState和enhancer都是可省略参数，所以要进行类型判断。
  当没有传入参数enhancer，并且preloadedState的值又是一个函数的话，createStore会认为你省略了preloadedState，因此第二个参数就是enhancer。
  再接下来判断enhancer，如果你传入了enhancer但是却又不是函数类型。会抛出错误。如果传入的reducer也不是函数，抛出相关错误。

  而后接下来是对变量的声明以及初始化

  和猜想的对比，发现多了两个变量，nextListeners和isDispatching。先猜想下这两个变量的用途。从名字上可以看出isDispatching应该是用在dispatch方法中作为一个标识，来确保在上一个dispatch方法执行结束前，下一个dispatch不会被执行。至于nextListeners暂时还看不出来。
  接着看下面的代码。

  ensureCanMutateNextListeners，用来确保nextListeners是currentListeners的一份拷贝。

  getState，不需要再解释了。

  subscribe

  开始依旧是类型检测判断，在这里又声明了一个标识符用来确保unsubscribe只有在第一次调用时才有用。当然，通过index是否大于-1来确保不会移除其他listener其实也没问题。一个耗费空间(使用额外的闭包变量来保存标识)，一个是耗费时间(每次调用都需要检索)。
  在这里能看出的一点是，每次订阅或者取消订阅操作的都是nextListeners。但是看看Redux关于该函数的注释可以得知，currentListeners是nextListeners的一份快照。目的是为了防止在dispatch调用时，subscribe和unsubscribe不会对当前执行产生副作用，在每trigger listeners时，被执行的都是快照内的listener。
  到这，就解释了为什么需要currentListeners和nextListeners两个变量了。currentListeners是用于dispatch时执行的快照，而nextListeners是subscribe实际操作对象。

  dispatch

  和开始的猜想基本一致，首先检查传入的action是不是纯对象，如果不是则抛出异常。在实际开发中我们会用到异步action，但是Redux只支持同步action和同步reducer，Redux在这里也提醒了我们如果需要用到异步action来处理业务，需要使用中间件进行扩展，这个在后面再具体分析。然后检测，action中是否存在type，不存在也给出相应的错误提示。
  在这里多了isDispatching标识符，刚开始觉得应该是为了确保一次dispatch用来避免reducer中可能会出现的调用dispatch而造成死循环的情况。但是listener也有调用dispatch的可能，不知道为什么没有把触发监听操作也放入到代码块中进行处理。
  最后，在每次dispatch时，都对listeners做一次快照。然后执行快照中的listener。

  replaceReducer

  和预料的一样用传入的reducer来替换之前的reducer。并且在替换前做类型检查。但是这里面多了一个dispatch({ type: ActionTypes.INIT })。
  那它是做什么用的呢？熟悉Redux使用的都知道，在Redux中，要求使用者在编写reducer时，对于未处理action直接返回旧的state。那么显然就是用于重置当前状态，触发视图刷新。一般用于热更新。

  observable

  Redux在createStore中还暴露了一个方法，不过文档中并没有提及。从代码看来，该函数调用后返回一个对象，调用对象上的订阅方法用于订阅当前应用state，并返回用于解绑该订阅的函数。

个人猜想

  • reducers (Object): 一个对象，它的值（value） 对应不同的 reducer 函数，这些 reducer 函数后面会被合并成一个。

  简单的说，combineReducers的主要作用就是将一个个小的reducer拼装成一个大的reducer用来处理具有相同结构的state对象。

applyMiddleware

个人猜想

  在Redux这里，中间件处于action与reducer之间，它提供的是位于 action 被发起之后，到达 reducer 之前的扩展点。放张图

  从图里可以清晰的看出Redux里面的中间件起了什么作用。在action被发起后，会依次被中间件处理，然后才会到达dispatch，最后传递到reducer中进行处理。

  从官方文档可以得知，applyMiddleware(…middlewares)返回了(Function) 一个应用了 middleware 后的 store enhancer。这个 store enhancer 的签名是 createStore => createStore，但是最简单的使用方法就是直接作为最后一个 enhancer 参数传递给 createStore() 函数。
  这意味着applyMiddleware是一个三阶函数，…middlewrares => createStore => (reducers, initialState) ，而文档中关于其参数的介绍：

  • …middlewares (arguments): 遵循 Redux middleware API 的函数。每个 middleware 接受 Store 的 dispatch 和 getState 函数作为命名参数，并返回一个函数。该函数会被传入 被称为 next 的下一个 middleware 的 dispatch 方法，并返回一个接收 action 的新函数，这个函数可以直接调用 next(action)，或者在其他需要的时刻调用，甚至根本不去调用它。调用链中最后一个 middleware 会接受真实的 store 的 dispatch 方法作为 next 参数，并借此结束调用链。所以，middleware 的函数签名是 ({ getState, dispatch }) => next => action。

总结

  对比分析猜想实现和源码不难发现，Redux源码并不多，也没有过多的奇淫技巧，而且代码中还有一大部分是对传入的参数的校验，这些严格的校验也是成熟的框架所必须的，也从体现了一些Redux的设计思想。当对Redux有了一定的了解后，很容易的可以猜测出内部的大致实现方式。同样的，当认真的阅读源码后，也更容易的能够理解Redux的构想，以及为什么会有各式各样的参数的限制。
